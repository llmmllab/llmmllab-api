# Session Report — 2026-05-21 / 2026-05-22

A two-day investigation across three repos (`llmmllab-api`, `llmmllab-runner`,
`mcp-server-web`) that touched on server-side tooling, MCP integration,
durable KV cache persistence, runner concurrency, deploy infrastructure,
and triage cleanup. This document records every shipped change, every
diagnostic that produced a load-bearing finding, and the open questions
left for follow-up.

## TL;DR

- **Server-side tool execution** (`web_search` / `web_fetch`) was rewritten
  end-to-end. Iteration cap, dedup, async timeouts, parallel execution,
  and three override knobs landed in the API. Then the inline path was
  retired in favor of an MCP server, complete with CI/CD.
- **KV cache durability** was overhauled. Saves used to fire only on
  LRU eviction; now they fire after every successful turn, on SIGTERM,
  and survive pod restarts via stable model-id-keyed filenames. Restore
  is now gated to only fire when the slot's in-memory KV is invalid —
  this fixed a critical bug where indiscriminate restore was *corrupting*
  the cache.
- **MTP speculative decoding** was disabled after evidence pointed at
  it as the root cause of universal n_past=32 invalidations. Prefill
  speed jumped 2.5× and continuation turns now hit cache at 99.8%+.
- **claude-cli body mutation** was identified with byte-precise
  certainty as the *one* source of remaining cache mismatches that
  persisted after all the runner-side fixes. claude-cli rewrites
  position [N-1] (the previous turn's last assistant message) every
  single request. Confirmed across 12 consecutive turns. Cost is
  negligible (~hundreds of tokens) because the divergence is at the
  *end* of the conversation, not the start.
- **96 issues + 5 PRs closed** across API + runner repos after a
  log-monitor false-positive audit.
- One **outage** caused by a deploy triggering pod restart on a node
  whose NVIDIA driver/library were out of sync. Resolved with a host
  reboot; documented mitigation.

## Day 1 — Server-side tooling rewrite + MCP migration

### Audit of `web_search` / `web_fetch` server-side execution

User reported duplicated searches and mid-generation stops on tool calls.
Investigation surfaced five concrete reliability bugs in
`tools/server_tool_executor.py` + `graph/nodes/server_tools.py`:

1. **No iteration cap.** `state.server_tool_iterations` was incremented
   but never inspected; the routing function only checked "does the last
   message have tool calls?". A model spinning the tool loop hit
   LangGraph's `recursion_limit=25` and surfaced as
   *"Sorry, I could not complete your request."*

2. **No deduplication.** Identical `(name, args)` tool calls within one
   turn both fired. Same query across iterations re-executed.

3. **No per-tool timeout.** `web_search` / `web_fetch` awaited the
   underlying tool with no `asyncio.wait_for` — one slow URL stalled
   the entire stream.

4. **Sequential execution.** Multiple tool calls in one turn ran
   one-at-a-time instead of via `asyncio.gather`.

5. **Override path was implicit and not opt-out-able.** Any tool literally
   named `"WebSearch"` got intercepted; no env kill-switch, no per-tool
   flag.

Shipped in `db9970f`:

- `config.SERVER_TOOL_MAX_ITERATIONS = 4` consulted by
  `make_should_continue_server_tools` — when cap is hit, route to END
  regardless of remaining tool calls.
- In-iteration dedup (`dedupe_tool_calls`) keyed on canonical
  `(name, args)`. Pascal/snake-case aliases fold together.
- Cross-iteration result cache (`state.server_tool_call_cache`,
  dict-union reducer). Repeated calls hit the cache and skip network.
- `asyncio.wait_for` around `web_search` (30s) and `web_fetch` (60s).
- `asyncio.gather` for distinct tool calls in one iteration.
- Three override knobs:
  - env `SERVER_SIDE_TOOLS_ENABLED` (default `true`)
  - per-request `X-Server-Side-Tools: false` header (Anthropic router)
  - per-tool `{"execute": "client"}` flag

19 new unit tests in `test/unit/test_server_tools.py`. Full unit suite
green: 334 passed, 3 skipped.

### Concurrent session / cache-churn fixes

User reported sessions taking turns and caches being invalidated under
load. Investigation surfaced two structural problems:

**Problem 1: Workflow cache rebuilt per session.**

Cache key was `(user_id, model_name, session_id)`. The session id was
in the key because `GraphBuilder` baked `X-Session-ID` into
`ChatOpenAI(default_headers=...)` at build time — sharing a cached
workflow across sessions would contaminate the slot LRU.

Fix in `10b1df2`:

- `X-Session-ID` set per-request via an `httpx.AsyncClient` event hook
  reading `_session_id_ctx` at outbound time. Each request runs in its
  own asyncio task with its own contextvar value.
- Cache key collapsed to `(user_id, model_name)`. N sessions on the
  same model now reuse one cached workflow.

**Problem 2: Anti-cache-locality runner ranking.**

`_select_runner` preferred the endpoint with the *fewest* active handles
for load-spreading. When the same model lived on multiple runners,
consecutive acquires bounced between them — no KV-cache reuse.

Fix in `10b1df2`:

- Added `_last_endpoint_for_model: Dict[str, str]` sticky pin.
- `_select_runner` consults it first; if the endpoint is healthy and
  still hosts the model and circuit isn't open, return it.
- Falls back to ranked spread when no pin exists or pin is ineligible.
- `acquire_server` records the pin on every successful create.

Result: every session targeting one model lands on the same llama.cpp
server and benefits from `cache_prompt` across sessions.

10 new tests across `test_workflow_cache_sharing.py` and
`test_runner_client.py::TestRunnerClientStickyEndpoint`. Full unit
suite: 352 passed, 3 skipped.

## Day 2 — KV cache durability + claude-cli mutation diagnosis

### Discovery: save-on-evict is too narrow

The runner's slot save logic only fired on LRU eviction. With ≤4
concurrent sessions and 4 slots, LRU never evicted — so save never
fired. Pod restart cold-start cost was high; OOM kills lost all
cached state.

### Aggressive save + restore design (commit `7a3806a`)

Three new triggers:

1. **Save after every chat completion.** Hook in `_stream_upstream`'s
   iterator finally (streaming) + the non-streaming path. Only fires
   on 2xx outcomes.
2. **Save on SIGTERM.** New `save_all_active_slots()` called from
   `app.py`'s lifespan teardown. Iterates every server's SlotLRU and
   saves each pinned session. Bounded by `terminationGracePeriodSeconds`
   (30s default).
3. **Restore eagerly on startup.** Already partially in place via
   `_save_file_exists` + lazy restore; now leveraged by the per-turn
   saves landing files in `/slots`.

### Bugs found during validation

Five distinct bugs surfaced as we tried to validate the save/restore
worked. Each had a clear empirical signature, was tracked to one root
cause, and was fixed with one commit.

#### Bug A — Colon in session_id (commit `cccb6bf`)

openclaw sessions use `pck:UUID` from the priority_middleware namespacing.
llama.cpp's `/slots/{id}?action=save` rejects filenames containing `:`
with HTTP 400. Every openclaw save attempt silently failed.

Fix: added `_sanitize_for_filename` that replaces `[^A-Za-z0-9._-]`
with `_`. Applied at the single `_slot_file_path` call site.

#### Bug B — CancelledError swallowing post-turn save (commit `d5997b1`)

claude-cli closes the SSE stream as soon as it sees the final event.
FastAPI cancels the generator; `await _save_slot(...)` inside the
finally block raises `asyncio.CancelledError`. That extends
`BaseException`, not `Exception`, so the save's own `try/except Exception`
was bypassed — silently. No save fired for any Claude Code turn.

Fix: detach the save into `asyncio.create_task` via a new
`_schedule_post_turn_save`. Module-level strong-reference set
(`_pending_saves`) keeps the task alive against garbage collection.
Save now runs to completion independent of the request's cancellation.

Symptom in logs: Claude Code (`38cc27ce`) had dozens of completed turns
with zero `Slot save` events; openclaw (`pck:*`) had save attempts
(rejected for reason A). Difference was the SSE-close-early behavior.

#### Bug C — `_stable_model_key` import (commit `18eac3b`)

The function was meant to return a stable model-derived filename
suffix that survives runner restarts. It imported
`from cache import server_cache` — but `cache.py` only exports the
`ServerCache` *class*, not a singleton. The import silently failed
(caught by a broad except), and the function fell back to using
`server_id` as the suffix.

Result: every pod restart produced *different* filenames for the same
session (because `server_id` is a per-process UUID). Cross-restart
restore never found its file.

Fix: change the import to `from app import server_cache` (matches the
pattern every other call site uses). After the fix, files use
`_Qwen3_6_27B_MTP.bin` as a stable suffix and survive pod restarts
correctly.

#### Bug D — Restore-when-warm corrupts the cache (commit `dd0fb85`)

THE big finding. The previous logic was "if a save file exists,
restore". Every continuation turn of an already-pinned session
triggered a restore from disk. That restore *overwrote* llama.cpp's
slot.cache_tokens metadata in a way that broke prefix matching —
the next request saw `n_past = 32`, llama.cpp logged "forcing full
prompt re-processing due to lack of cache data", and erased all
internal context checkpoints.

Confirmed empirically:

| Task | Behavior | n_past result |
|---|---|---|
| 252 | save → launch (no restore) | **n_past = 68,347 (99.7% hit)** ✓ |
| 348 | save → **restore** → launch | **n_past = 32 (full re-prefill)** ✗ |

Same slot, same session, KV unchanged between save and restore.
The *only* difference was an unnecessary restore call.

Fix: only restore when the slot's in-memory KV cannot be trusted:

- `evicted_pair is not None` — LRU just reassigned the slot from
  another session
- `session_id not in _seen_session_on_server[server_id]` — first
  request for this (server, session) pair since process start

Steady-state continuation turns now skip the restore entirely. New
module-level set `_seen_session_on_server` tracks per-server which
sessions have current in-memory state; entries are removed on LRU
eviction.

#### Bug E — Speculative decoding (`--spec-type draft-mtp`) broke cache continuity (commit `1aa54be`)

Earlier in the session I'd incorrectly *dismissed* spec decoding as a
cause. Later evidence proved it was a major contributor. With
`--spec-type draft-mtp` enabled, **every** turn (post-restore) gave
`n_past = 32` followed by *"forcing full prompt re-processing due to
lack of cache data"*. Internal context checkpoints existed but were
unusable.

Fix: commented out `spec_type_mtp` and `spec_draft_n_max` in
`.models.yaml` for the `Qwen3_6_27B_MTP` model. Re-enable by
uncommenting both lines.

Immediate measured effects:

- Prefill speed **2.5×** faster (485 t/s → 1,160-1,250 t/s)
- Cache hit warnings dropped from one per turn to one per slot
  (cold-start only)
- `n_past = 32` events: from every turn → only on llama-server
  process recycle

Trade-off: generation speed drops back to baseline (no spec
acceleration). For long-context coding workloads (40k+ token prompts,
short generations), the prefill speedup + cache continuity dwarfs
the generation loss. Net throughput is much higher.

### Cleanup thresholds were too aggressive (env-var tuning, deploy-side)

The runner's slot-cleanup task runs every 5 min and was deleting
save files for sessions idle 12+ minutes. On a 264 GB-free filesystem,
the cap was set to 5 GB. Interactive coding sessions easily go 30+
minutes between turns, so by the time the user typed again the save
file was gone and the session paid full cold-start cost.

Recommended deployment-side changes (not yet shipped — config-only):

```bash
SLOT_INACTIVE_MAX_AGE_MIN=240   # 4 hours
SLOT_CLEANUP_MAX_AGE_MIN=240    # 4 hours
SLOT_CLEANUP_MAX_SIZE_MB=80000  # 80 GB cap
```

## The claude-cli mutation diagnosis

After all five bugs above were fixed, n_past=32 events were rare but
still happened. The remaining question: is claude-cli mutating its
own conversation between turns, or is the API doing something subtle
we haven't found?

To get **absolute byte-level certainty**, we layered four
fingerprint signals into the API:

1. **`wire_hash_full`** — SHA256 of the raw HTTP body bytes, read
   via `await request.body()` BEFORE FastAPI parses anything. Immune
   to any Python-side transformation. Commit `9c0b790`.

2. **`raw_per_message_hashes`** — per-message canonical-JSON hashes
   computed *before* `_strip_server_tool_blocks` runs. Commits
   `cf71e42` and `9c0b790`.

3. **`per_message_hashes`** — same, computed *after* the strip.
   Comparing the two reveals if the strip is the cause of any
   message-level drift.

4. **`system_hash` / `tools_hash`** — separate fingerprints for the
   stable session-scoped fields.

### Result

Across **12 consecutive 38cc Claude Code turns**, with full
fingerprint coverage on every one:

```
Every comparison N → N+1:
  system_hash         MATCH (d2b1f1f420a5 stable)
  tools_hash          MATCH (ae0268e1e862 stable)
  bytes_removed_strip 0 in both (strip is a no-op)
  Positions [0..N-2]  byte-identical raw per-message hashes
  Position [N-1]      DIFFERS
  Position [N..end]   new content (expected)
```

**With absolute, byte-precise certainty: claude-cli rewrites the
previous turn's last assistant message between every turn.** Most
likely the streaming SSE `content_block_delta` events get
reconsolidated into a canonical final form for claude-cli's local
history — and that canonical form differs in some byte-level detail
from the raw stream it received from us.

### Why openclaw doesn't have this problem

openclaw uses OpenAI's `/v1/chat/completions` protocol. There's no
stream-event-vs-consolidated-response transformation; the assistant
response goes back to the API exactly as it was received. `pck:*`
sessions don't have position-[N-1] mutation, so cache prefix-matching
goes much further before diverging.

### Why this isn't a big problem in practice

Position [N-1] is the **last** message of the prior conversation —
deep into the prompt. With the runner-side fixes (save-after-every-turn,
skip-restore-when-warm, spec-decoding off), cache hits for everything
up to position [N-1]. The post-restart steady-state data shows:

| Turn | n_past | Total tokens | Hit % |
|---|---|---|---|
| 23:47:10 | 89,769 | 89,881 | **99.9%** |
| 23:47:54 | 90,229 | 90,423 | **99.8%** |
| 23:49:54 | 94,658 | 94,855 | **99.8%** |
| 23:50:26 | 94,955 | 95,126 | **99.8%** |
| 23:50:54 | 95,215 | 95,376 | **99.8%** |

The position-[N-1] mutation costs ~150-200 tokens of re-prefill per
turn — essentially invisible against a 95k-token cached prompt.

### What we still can't prove

A handful of `n_past = 32` events occurred *before* the byte-level
fingerprint instrumentation was live. Some of these correlated with
llama-server process restarts (server eviction/recycle by the runner's
ServerCache) — those are expected cold-starts. But some may have been
claude-cli mutating *earlier* messages too (mid-conversation finalization
of a tool-use chain, for instance).

Open question for follow-up: instrument the diff on a turn that
triggers `n_past = 32` while fingerprints are live. The current
instrumentation will catch it.

## Outage and recovery

When I pushed runner commit `ca12827` for new instrumentation, the
existing runner pod got rolled. The new pod failed to start with:

```
failed to fulfil mount request:
open /run/nvidia-persistenced/socket: no such file or directory
```

Diagnosed as a host-level NVIDIA driver/library version mismatch on
lsnode-3 — userspace `libnvidia-ml.so` had been updated to 580.159
while the kernel modules still ran an older version. The previously
running pod survived because its container's mount namespace was
established before the upgrade. My deploy was the trigger that
required a fresh mount, which the host could no longer provide.

Recovery:

```bash
sudo reboot   # (preferred — user did this)
# or, surgical:
# 1. kubectl drain lsnode-3 --ignore-daemonsets --delete-emptydir-data --force
# 2. sudo systemctl stop nvidia-persistenced
# 3. sudo rmmod nvidia_uvm nvidia_drm nvidia_modeset nvidia
# 4. sudo modprobe nvidia nvidia_modeset nvidia_drm nvidia_uvm
# 5. sudo systemctl start nvidia-persistenced && sudo systemctl enable nvidia-persistenced
# 6. nvidia-smi   # verify
# 7. kubectl uncordon lsnode-3
```

Mitigation for the future:

```bash
sudo apt-mark hold nvidia-driver-* libnvidia-* nvidia-utils-* nvidia-kernel-*
```

Lesson learned: any runner pod restart on a node with a latent
driver-version mismatch will fail at OCI runtime. Worth probing
`nvidia-smi` health before deploying runner changes that trigger
a rollout.

## Instrumentation now live

For future investigation, the following structured log events are
emitted on every relevant action:

### API side

- **`Anthropic body fingerprint`** (`routers/anthropic/messages.py`):
  `wire_hash_full`, `wire_bytes`, `raw_hash_full`, `raw_hash_8k`,
  `stripped_hash_full`, `stripped_hash_8k`, `strippable_block_counts`,
  `msg_count_in`, `bytes_removed_by_strip`, `raw_per_message_hashes`,
  `per_message_hashes`, `system_hash`, `tools_hash`.

- **`Runner request fingerprint`** (`graph/workflows/base.py`):
  `body_bytes`, `hash_full`, `hash_8k`, `hash_256` — what we sent to
  the runner.

### Runner side

- **`Slot prefill complete`** (`server_manager/base.py`): per-turn
  prefill metric extracted from llama.cpp's `print_timing` line.
  `task_id`, `n_processed`, `prefill_seconds`, `prefill_tokens_per_sec`.
  Fires every turn; small `n_processed` = cache hit.

- **`Slot cache-hit per-turn`** (same file): when llama.cpp logs the
  warn-branch `n_past`/`slot.prompt.tokens.size()` line, emit a
  structured event with both values + `cache_hit_pct`.

- **`Slot save` / `Slot restore`** (`proxy/router.py`): one log
  line per save/restore call with `session_id`, `slot_id`, `status`,
  `n_saved` / `n_restored`, `slot_file`.

- **`Resolved slot via LRU`** (same file): `session_id`, `server_id`,
  `slot_id`, `evicted` (the evicted session if any).

Reading these together gives a complete picture of cache behavior from
"what claude-cli put on the wire" through "what KV state ended up on
disk" in every direction.

## Open follow-up items

These were identified but not shipped this session:

1. **Cleanup-threshold tuning** — the runner's slot file cleanup is too
   aggressive (12-min idle threshold, 5 GB cap on a 264 GB filesystem).
   Recommended bumps documented above.

2. **n_past=32 root-cause completion** — a few invalidations remain
   unattributed (happened before byte-level fingerprint coverage). Next
   such event with full fingerprint coverage will resolve it. The
   instrumentation is ready and waiting.

3. **`messages_from_anthropic` 1:N expansion auditing** — the
   Anthropic-to-internal conversion does tool_result block expansion
   that the OpenAI path doesn't. If the residual cache misses persist
   after pinning down claude-cli's behavior, audit this conversion for
   non-determinism.

4. **Issue #11** — Plan B: patch llama.cpp scheduler for per-slot
   batch quota. Deferred since first day; still pending.

5. **Failing tests in CI** — the test-suite identification task
   (#15-#17) remains pending across both repos.

6. **Per-server-process slot save on eviction** — the SIGTERM handler
   saves slots on pod shutdown, but the runner's internal ServerCache
   eviction (recycling individual llama-server processes after
   `EVICTION_TIMEOUT_MIN`) does *not* trigger a save. This is why we
   still see `n_past = 32` correlated with server-process recycling
   — the recycled process loses its slot state with no disk backup
   from the recycle event. Adding a save hook to the eviction path
   would close this gap.
